### 1. UI / UX 란?

- **UI : 디자인**
  사용자가 서비스를 사용할 때 마주하는 디자인. 눈에 보이는 것!
  ex) 폰트, 컬러, 간격, 레이아웃 등

- **UX : 경험**
  서비스를 사용하는 사용자가 갖는 다양한 경험 (감정)

![UIUX 예시](/images/blog/uiux.png)
‘하인즈 케첩’으로 쉬운 예를 들 수 있다. <br />
초기 하인즈 케첩은 유리병 형태였다고 한다. 사용자들은 케첩을 사용하면서 _1) 사용할때마다 케첩이 사방으로 튀고 2) 끝까지 케업을 다 사용하지 못하는_ 불편함을 경험하게 된다.
예쁘기만 했던 초기의 유리병 형태의 케인즈 케첩이 지금의 형태로 변경된 데에는 사용자들의 지속적인 피드백을 관찰하여 더 나은 UX를 제공하기 위함일 것이다.

- 목적 : 사용자가 편리하도록 UI를 디자인해서 더 나은 사용자 경험을 제공하는 것.

### 2. UX를 왜 신경써야 할까?

- 복잡해진 웹
  웹이 발전함에 따라 정보를 표현할 수 있는 방식이 다양해졌다.<br />
  커뮤니케이션에도 청자를 위해 듣기 쉬운 언어로 말해야 하듯이, **프로덕트 개발 또한 사용자 입장에서 쉽게 다가갈 수 있도록 만들어야 한다.**

### 3. UX 이야기

1. 제이콥의 법칙 : 사용자는 새로운 경험을 이해하기 위해 기존 경험을 활용한다.<br />
   ex) 장바구니의 위치 / 보통의 서비스들에는 우측 상단에 장바구니가 위치해있다.

2. 피츠의 법칙 : 터치 대상의 크기 / 사이에 충분한 거리 확보
   → 권장되는 규격이 있다.

3. 힉의 법칙 : 의사 결정에 걸리는 시간은 선택지의 개수와 복잡성에 비례해 늘어난다. <br />
   선택지의 개수를 최소화하는 것이 좋음!
   ex) 할아버지 할머니 전용 리모컨

4. 피크엔드 법칙 : 절정의 순간과 마지막 순간에 느낀 감정을 바탕으로 경험을 판단하는 경향이 있음 <br />
   → 부정적 절정 : 404 페이지 <br />
   → 404 페이지에 약간의 유머를 활용하면? 오히려 더 즐거운 경험으로 바꿀 수 있는 기회이다.

5. 폰 레스토프 효과 : 비슷한 사물이 여러개있으면 그중에서 가장 차이가 나는 한가지만 기억할 가능성이 크다. <br />
   → 강조할 것들은 정확히! 강조해야함

6. 도히티 임계 : 시스템의 반응 속도는 전체 사용자 경험을 좌우하는 중요한 요소이다. <br />
   → 규정한 시간은 0.4초 / 더 많이 걸려도 개선할 수 있는 방법이 없을땐? <br />
   → 사용자에게 처리 시간에 관한 **시각적인 피드백**을 준다. <br />
   피드백이 있으면 : 사용자는 대기시간에 더 관대해지고, 실제 웹이 더 빠르게 작동한다고 느낀다.

q. 모든 상황에서 로딩이 빠른 것이 좋을까? <br />
No! <br />
사용자 기대보다 반응이 너무 빠르면 → 눈치조차 채지 못할 수 있다. <br />
예를들어 보안 점검 같은건 의도적으로 작업 완료를 늦게해야 사용자가 안심하고 보안 점검의 결과를 믿을 수 있게된다.

### 4. 프론트엔드 개발자와 UX 이야기

프론트엔드 개발자가 왜 UX를 신경써야 할까? <br />
개발자는 사용자가 쉽고 편안하게 사용할 수 있는 서비스를 제공해야 한다. <br />
물론 UX 디자이너가 되어야 한다는 것이 아니다. 언젠가 만날 훌륭한 디자이너와
그룹이 아닌 팀으로 협력하기를 바라는 것이다. 그렇기에 참여하는 프로젝트를
성공적으로 만들고 싶다면 최소한의 UX의 기본에 관심을 가져야 하는 것이다.

결국은 서비스가 잘 되는 것이 목표이다. 본질은 좋은 서비스를 제공하는 것!
